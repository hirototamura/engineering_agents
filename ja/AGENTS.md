# AGENTS.md — Engineering Agents 向けガイド

> English: [../en/AGENTS.md](../en/AGENTS.md)

本ファイルは、**人間のコントリビュータ**と**コーディングエージェント（Cursor 等）**の双方に向けた、リポジトリの北極星と実装規律を定義する。

---

## ミッション

複雑なハードウェアを**自律開発**するための**設計検証ループ**を回すエンジニアチームをソフトウェアで再現する。

対象は**人工衛星・宇宙船・ロケット・航空機・自動車・ヒューマノイドロボット**などの従来システムズエンジニアリングを必要とするハードウェア全般である。高忠実度 3D や実機接続より、**再現可能な設計→検証ループ**を優先する。

**本リポジトリの主たるスコープは設計と検証である。** 監督（設計要求・検証要求の設定と、その要求自体の妥当性確認）は別リポジトリ **[One Piece](https://github.com/hirototamura/one-piece)** が担う。最終的には One Piece から要求を取得する連携が必要になるが、**現時点では実装しない**。本リポジトリでは `scenario.yaml` 等で要求をスタブし、設計・検証ループの実装に集中する。

## パラダイムシフト

ビジョンを示す表であり、時期や比率の予測を目的としない。


| 段階       | 監督      | 設計  | 検証                             |
| -------- | ------- | --- | ------------------------------ |
| **現在**   | 人間      | 人間  | 仮想世界：物理シミュレーション + 人間 物理世界：人間   |
| **10年後** | 人間 + AI | AI  | 仮想世界：物理シミュレーション + AI 物理世界：ロボット |
| **本デモ**  | 人間 + AI | AI  | 仮想世界：物理シミュレーション + AI 物理世界：N/A  |


**現在** → **本デモ** で、設計と仮想検証の主役が人間から AI へ移る。**本デモ** → **10年後** で、監督に AI が加わり、物理世界の検証が人間からロボットへ移る。

### 監督・設計・検証 — 要求がつなぐ三者

監督（人間 + AI）が **設計要求** と **検証要求** の両方を与える。監督たるものは、この二種類の要求を満たすフレームを用意しなければならない。

### 最重要要求 — ミッションを侵してはならない

すべての要求の上位に、**最重要要求**がある。これは**ミッションそのもの**（何のために存在するか）であり、現実や物理法則とは別軸である。**決して侵してはならない**。設計要求・検証要求は反復改訂できるが、ミッションを放棄・すり替えする行為は許されない。


| 種類            | 改訂       | 例                          |
| ------------- | -------- | -------------------------- |
| **最重要要求**     | 不可       | ミッションそのもの                  |
| **設計要求・検証要求** | 監督が反復改訂可 | 閾値・トポロジ前提・受入シナリオ（初期案は誤りうる） |


ミッションを達成するために従うべきは、まずこの**最重要要求**である。一方、**現実（物理法則）** は設計・検証が守るべき制約であり、検証は物理世界の知覚としてそれを問う。監督の設計要求・検証要求が物理と矛盾するとき、設計をねじ曲げるのではなく**要求の改訂を監督に提案**できる — その提案もミッション（最重要要求）を損なってはならない。


| 役割     | 本質       | 満たすべきこと                                                                      |
| ------ | -------- | ---------------------------------------------------------------------------- |
| **監督** | 要求の設定    | 設計に対する要求と、検証に対する要求を与える                                                       |
| **設計** | 要求に沿った構想 | 監督の**設計要求**を満たし、**物理の原理**に沿った設計提案を出力する。要求が誤っている場合は**要求の変更を監督に提案**する          |
| **検証** | 物理世界の知覚  | 設計提案に基づきシミュレーション（仮想）または観測（物理）を行い、監督の**検証要求**を満たすかを確認する。必要なら**要求の変更を監督に提案**する |


検証は物理法則のシミュレーションやロボットによる観測であり、**物理世界の知覚**である。それに加え、監督が定めた検証要求（閾値・シナリオ・受入基準）への適合確認が不可欠である。仮想世界では、設計提案を入力として物理シミュレーションを回し、監督の検証要求に照らして合否を決める。

監督の設計要求・検証要求は初期段階で**誤っていることがある**。そのとき設計だけをねじ曲げて適合させる必要はない。**設計要求・検証要求そのものを改訂する**方が重要な場合がある。設計・検証は監督（One Piece）に対し変更を提案でき、三者はこれらの要求を何度でも反復する — ただし**ミッション（最重要要求）を損なわない範囲でのみ**。

```text
最重要要求（ミッションそのもの — 侵さない）
  ↓
監督 ⇄ 設計要求 + 検証要求（反復改訂。正本は One Piece）
  ↓
設計: 設計要求を満たす設計提案（物理原理に整合）
      または 監督への要求変更提案（ミッションを損なわない範囲で）
  ↓
検証: 設計提案 → 物理世界の知覚（シミュレーション／観測）
      → 検証要求を満たすか（決定論的チェック）
      または 監督への要求変更提案（ミッションを損なわない範囲で）
```

ループの出口は「設計を変えて要求に合わせる」だけではない。**設計要求・検証要求を変えて現実（物理）に合わせる**ことも正当な成果である。物理に反する設計や、それを隠す検証は禁止する。いっぽう**ミッション（最重要要求）を放棄する**設計・検証も、いかなる要求改訂の名目でも禁止する。

物理世界の検証は本デモでは **N/A**（表のとおり）。本リポジトリが実装するのは **仮想世界** のみ — **Space Station OS（SSOS）をモックした物理シミュレーション** に設計を投入し、**AI（検証ブリッジ）** が決定論的チェックまで回す。実機 ROS2 / 軌道上システムへの接続はスコープ外（`SsosAdapter` はスタブ）。

## 最重要原則: 自作自演の禁止

表の **10年後** および **本デモ** では、設計も仮想世界の検証も AI が担う。ここが一歩間違えると **AI の自作自演（ハルシネーションの隠蔽）** になる。


|              | 本デモ・10年後           | 自作自演にならない条件                                                 |
| ------------ | ------------------ | ----------------------------------------------------------- |
| **監督**       | 人間 + AI            | **設計要求**と**検証要求**の両方を与える。AI は状況判断・議論に留まり、要求の最終設定は人間が担う      |
| **設計**       | AI                 | 設計提案を出力する。要求が物理原理と矛盾する場合は**要求変更を監督に提案**する（合否判定・要求の独断変更はしない） |
| **検証（仮想世界）** | 物理シミュレーション + AI    | シミュレーション結果に基づき検証要求を確認する。テレメトリが要求の妥当性を問う場合は**要求変更を監督に提案**する  |
| **検証（物理世界）** | 本デモ N/A ／ 10年後ロボット | ロボットによる物理世界の知覚 + 検証要求の確認。仮想合格を物理合格の代替とみなさない                 |


信頼できるエンタープライズ製品にするには、**創造性（設計の AI）** と **物理法則の冷徹さ（シミュレーション + 決定論チェック）** を分離し、仮想世界の合否を LLM の主観から独立させる。**ミッション（最重要要求）の放棄**と、**物理に反する設計・検証**のいずれも禁止する。

### 監督（人間 + AI）が担うこと

> **スコープ注記:** 監督の実装は **One Piece** リポジトリが担う。本リポジトリでは要求を `scenario.yaml` でスタブするのみ。One Piece からの要求 pull は将来課題。

- **設計要求の設定** — 満たすべき性能・制約・トポロジの前提（何を作るか）
- **検証要求の設定** — 受入基準・閾値・シナリオ（何をもって合格とするか）
- **人間**: 上記要求の定義・改訂の最終判断（設計・検証からの要求変更提案を受け入れるか）
- **AI**: チーム内の議論・状況判断・要求の解釈補助（要求そのものの決定権は人間）

### 設計（AI）が担うこと

- 監督の**設計要求**を満たすトポロジ・パラメータ・恒久変更の**提案**（`design_proposals.json` 等）
- **物理の原理**に沿った設計提案の出力（ソルバーに投入可能な形式）
- 要求が非現実的・矛盾する場合、**監督への設計要求・検証要求の変更提案**（設計をねじ曲げるより要求を直す）
- ランタイム中の回復コマンド案の生成

### 検証が担うこと — 仮想世界（物理シミュレーション + AI）

1. **物理世界の知覚** — 設計提案を入力に、確定的な物理ソルバー（SSOS の数理エンジン）でシミュレーションを実行する。解釈・丸め・「たぶん大丈夫」は禁止。
2. **検証要求の確認** — シミュレーションの生データ（テレメトリ JSON）が、監督が設定した**検証要求**（ハード制約・閾値・シナリオ）を満たすか、決定論的プログラムで判定する。LLM に pass/fail を聞かない。
3. **要求変更の提案** — シミュレーション結果が要求の誤りを示す場合、監督に検証要求（必要なら設計要求も）の改訂を提案する。合否の根拠はテレメトリであり、要求の改訂決定は監督が行う。
4. **物理世界は別系統** — 本デモでは N/A。10年後はロボットが物理世界の知覚と検証要求の確認を担う想定だが、仮想合格は物理合格の十分条件ではない。

## 本リポジトリでの具体化

本デモは表の **本デモ** 行のうち **設計** と **検証（仮想世界）** を実装する。ECLSS/EPS は参照シナリオに過ぎず、パラダイムはドメイン横断である。

表の **監督** 列は One Piece 側の責務である。現行 MVP では `scenario.yaml` の閾値・シナリオが検証要求のスタブであり、設計要求のスタブも同ファイル周辺に置く。One Piece から要求を pull する連携は将来課題とし、今は手を入れない（[one-piece-integration.md](docs/one-piece-integration.md) は現状 provenance エクスポートのみ）。

### 現行 MVP（`scrubber_degradation`）— 表との対応


| 表の列             | 本デモでの主体                                | 出力・実装                                     | 備考                                          |
| --------------- | -------------------------------------- | ----------------------------------------- | ------------------------------------------- |
| **監督（人間 + AI）** | One Piece（将来）／現状は `scenario.yaml` でスタブ | `scenario.yaml`, `messages.jsonl`         | 本リポジトリの主スコープ外。要求の正本は One Piece へ移す予定        |
| **設計（AI）**      | 代表エンジニア（LLM またはルール）                    | `design_proposals.json`                   | 設計要求への適合 + 物理原理に沿った設計条件。ランタイム中はシミュレータに適用しない |
| **検証・仮想世界**     | 物理シミュレーション + 検証ブリッジ                    | `telemetry.jsonl`, `health_metrics.jsonl` | 設計提案を入力にシミュレーション → 検証要求を決定論的に確認             |
| **検証・物理世界**     | N/A                                    | —                                         | 本リポジトリのスコープ外                                |


ダッシュボードの **After (if proposals applied)** は提案のプレビューであり、**仮想世界の検証済み結果ではない**。設計をシミュレータに再投入して閉ループするパイプラインは、本デモ行を完成させる次段階である。

### 目指す完成形（本デモ行 → 10年後への土台）

```text
[最重要要求] ミッションそのもの（侵さない。One Piece が定義）
        ↓
[監督] One Piece（将来）— 設計要求 + 検証要求の設定・改訂
       ⇄ 設計・検証からの要求変更提案（ミッションを損なわない範囲）
       現状: scenario.yaml でスタブ（本リポジトリは未連携）
        ↓
[設計] AI（LLM）— 本リポジトリの主スコープ
  → 設計提案（`design_proposals.json` — scrubber は `add_edge` 等 / ssos_eclss_loop は `design_domain: ssos_graph`）
  → 必要なら監督への要求変更提案
        ↓
[検証・仮想世界] 物理シミュレーション + AI（検証ブリッジ）
  → 設計提案を入力に物理世界の知覚（シミュレーション）
  → telemetry.jsonl（生データ）
  → 検証要求チェッカー（純 Python、閾値比較）
  → pass / fail（決定論的）
  → 必要なら監督への要求変更提案
        ↓
[検証・物理世界] 本デモ: N/A ／ 10年後: ロボットによる知覚 + 検証要求確認
```

検証ブリッジに LLM の合否判定を挟まない。説明文を LLM が書くことはあっても、**合否根拠はシミュレーション出力と検証要求の差分**に置く。

## コーディングエージェント向け規律

### レイヤと依存方向（厳守）

```text
tools → scenario → environment → core
integrations/one_piece  ← scenario から呼び出し
```

下位レイヤから上位を import しない。`environment` に LLM や Persona を入れない。


| パス                  | 責務                                            |
| ------------------- | --------------------------------------------- |
| `src/core/`         | Persona、Team、メモリ、LLM クライアント、イベントログ            |
| `src/environment/`  | `SimulatorProtocol`、ECLSS/EPS 物理モック、SSOS トピック |
| `src/scenario/`     | シナリオ YAML、runner、チーム実装                        |
| `src/tools/`        | Streamlit ダッシュボード                             |
| `src/integrations/` | One Piece provenance                          |


### やってよいこと

- 表の **本デモ** 行のうち **設計** と **仮想検証** に集中する。監督の要求は `scenario.yaml` でスタブし、One Piece 連携は今は着手しない。
- **設計要求**と**検証要求**を `scenario.yaml` 等で明示し、版管理する（One Piece 連携までの暫定スタブ）。
- 設計・検証が監督に**要求変更を提案**する経路を設計する（監督が改訂するまで現行要求で判定する）。
- 要求が非現実的なとき、設計だけを曲げて適合させるのではなく**設計要求・検証要求の改訂**を検討する（**最重要要求**は侵さない）。
- 検証は設計提案を入力にシミュレーションを回し、**現行の検証要求**を決定論的に確認する。
- 設計提案とランタイム操作の**分離**を維持する（現行: 回復コマンドのみランタイム適用、恒久変更は事後提案）。
- `labeled_rule_base` の `policy` と `llm` モードの**隔離**を維持する（LLM プロンプトに policy 閾値を混ぜない）。
- 仮想世界の合否判定を `**health_metrics` / シナリオ YAML / 純関数チェッカー** に追加する。
- シミュレータ出力は **JSONL スキーマ**（[docs/api-contracts.md](docs/api-contracts.md)）に従う。
- 変更後は `pytest` を実行する。

### やってはいけないこと

- **ミッション（最重要要求）**を放棄・すり替えする設計・検証を行う。
- **物理に反する**設計を提案する、または検証でそれを隠す。
- 設計要求・検証要求なしに設計・検証ループを回す（監督不在の自作自演）。
- 設計・検証が**要求を独断で書き換える**（変更提案は監督へ。正本は One Piece / `scenario.yaml`）。
- 物理知覚（シミュレーション）なしに検証要求の充足を宣言する。
- 仮想世界の「物理シミュレーション + AI」を、**LLM の主観による合否**で実装する（自作自演）。
- 検証ブリッジがシミュレータをバイパスし、設計内容から「期待どおり」と推論する。
- 物理世界が N/A の本デモで、仮想合格を**物理検証済み**と表現する。
- ランタイム中に恒久トポロジを変え、設計と検証の分離を壊す（`apply_design_change` は Phase 0 で削除済み）。
- `environment` にエージェントロジックや Ollama 依存を入れる。
- テレメトリや閾値を「だいたい」で丸めて合格扱いにする。

### エージェントモード（`agents.mode`）


| モード                 | 用途                 |
| ------------------- | ------------------ |
| `none`              | ベースライン（エージェントなし）   |
| `labeled_rule_base` | 再現性の高い正解比較・回帰テスト   |
| `llm`               | 状況判断・議論・設計提案の多様性実験 |


`labeled_rule_base` は検証パイプラインの**足場**（監督・設計の比較用）。`llm` は設計・監督の AI 側実験。いずれも仮想世界の物理検証はシミュレータが担う。

### テストと実行

```bash
pip install -e ".[dev]"
pytest
python -c "from scenario.runner import run_scenario; print(run_scenario('scrubber_degradation', overrides={'agents': {'mode': 'labeled_rule_base'}}))"
```

LLM モードは Ollama 必須。CI・回帰は Fake LLM / `labeled_rule_base` を優先する。

## 要約（エージェントへの一行）

**最重要要求はミッションそのもの。設計要求・検証要求は One Piece 下で反復改訂できる。設計と検証は物理に従い、ミッションを侵さない。**
